# AIコーディング体験記 — morning-notify 開発を通じて

## はじめに

このドキュメントは、`morning-notify`（毎朝Discordに天気・カレンダー・ニュースを通知するPythonアプリ）の開発を通じて得た、AIコーディングツールとの協働体験を記録したものです。

使用ツール：**Claude Code**（Anthropic）

| 項目 | 内容 |
|------|------|
| 期間 | 2026年4月 |
| 使用言語 | Python 3.11 |

| 通知先 | Discord Webhook |

## 開発の流れ

## やってみてわかったこと

### よかった点

**日本語の曖昧な指示でも動く**

「もう少しわかりやすく」「専門的じゃない」といった抽象的な言葉でも、文脈から意図を汲んで適切な実装を提案してくれた。仕様書を書く必要がなかった。

**コードだけでなくGit操作・CIも任せられた**

ブランチ作成・コミット・プッシュ・PR作成まで一貫して対応してくれた。「新規ブランチを切ってPRを作る」というルールを一度伝えたら、以降は自動的にその作法に従ってくれた。

**エラーをそのまま貼るだけで直してくれた**

CI が落ちたときにエラーログをそのまま貼り付けると、原因を特定して修正してくれた。自分でデバッグする時間が大幅に短縮された。

**スマホだけで開発できた**

PCなしでスマホから操作しながら、コードの追加・修正・PRの作成が完結した。唯一PCが必要だったのは Google OAuth のリフレッシュトークン取得だけ（ブラウザが必要なため）。

**実装前に選択肢を提示してくれた**

ニュースソースの変更や認証方式など、複数の選択肢がある場面では「こういうアプローチがある、おすすめはこれ」と整理して提示してくれた。自分で調べる手間が省けた。

### 注意が必要だった点

**PRのコンフリクト**

複数のPRが並走したとき、マージ順序によってコンフリクトが発生した。AIはその都度「クローズして新しいPRを作る」「リベースする」などの対応を提案してくれたが、ブランチ管理は自分でも意識する必要があった。

**意図と実装がずれることがある**

「セキュリティ、政治、経済を追加して」と伝えたとき、政治・経済がすでに実装済みであることをAIが先に確認して教えてくれた。依頼の前に現状を把握しきれていないことがある。

**AIが知らない情報には答えられない**

「このプラグインを入れて」という（偽）情報を渡したとき、存在しないプラグインとして正しく指摘してくれた。AIは知識外の主張には慎重に対応する。

**外部サービスの実態はAIでは確認できない**

NHKのRSSが取れているかどうかは、実際に動かさないとわからなかった。コードは正しく書けても、外部APIの挙動は本番環境でしか確認できない部分がある。

## AIへの指示で効果的だったパターン

```
# 良かった指示の例

「〇〇を追加してほしい。プルリクも作って、テストコードも更新して」
→ 実装・テスト・PRまで一度にやってくれる

「なんかエラー出てます → [エラーログ貼り付け]」
→ 原因特定と修正まで対応してくれる

「〇〇の選択肢ってある？」
→ 複数案とトレードオフを整理して提示してくれる

「先行して実装できる？（Secretsがまだない状態で）」
→ 「Secretsがなくてもフォールバックする設計にする」など現実的な対応を提案してくれる
```

## 開発ルールとして定着したこと

このプロジェクトを通じて、以下を CLAUDE.md に明文化した。

- **作業は必ず新規ブランチを切る**（`claude/<作業内容>` 形式）
- **実装が完了したらPRを作成する**
- **masterへの直接コミットは禁止**

一度ルールを明文化すると、以降は自動的にその作法に従ってくれた。

## まとめ

「自分でコードが書けなくても、何を作りたいかが明確であれば開発できる」という体験ができた。一方で、**何を作りたいかを言語化する力** と **AIの出力を検証する目** は依然として人間に求められる部分だと感じた。

AIはコードを書く速度を大幅に上げてくれるが、「何を作るか」「それが正しく動いているか」の判断は自分でする必要がある。ツールとしての位置づけを正しく理解した上で使うことで、一人でも想定以上のものが作れると実感した。
