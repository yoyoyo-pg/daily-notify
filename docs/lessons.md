# Self-Improvement Rules（自己改善ルール）

セッション開始時に必ず確認する。ユーザーから指摘を受けたら即座に追記し、同じミスを繰り返さないための再発防止ルールを書く。

**フォーマット**
```
### [ルール名（動詞で始まる命令形）]
**ルール**: 具体的に何をする/しない
**なぜ**: 過去に起きた問題・インシデント
**適用条件**: どのときに発動するか
```

---

## Gitブランチ管理

### `gh pr list` は必ず `--state all` を付ける
**ルール**: `gh pr list` 単体はオープンPRのみ表示。必ず `--state all` を付ける。  
**なぜ**: マージ済みブランチを見落とし、そのブランチに追加コミットしてもPRに反映されなかった。  
**適用条件**: PRの状態確認・既存ブランチを再利用しようとするとき

### 作業前に `gh pr view <branch>` でブランチ状態を確認する
**ルール**: 「このブランチで続けよう」と思っても、先にPR状態を確認してから着手する。  
**なぜ**: マージ済みブランチへの追加コミットはPRに反映されず、作業が無駄になる。  
**適用条件**: 既存ブランチを再利用するとき

### rebase 後の force push はユーザーに確認を取ってから実行する
**ルール**: `git push --force-with-lease` は実行前に必ずユーザーの許可を得る。  
**なぜ**: `settings.json` の deny リストに `git push --force*` があり、迂回は意図に反する。  
**適用条件**: rebase 後に push が必要になったとき

### `.claude/settings.json` はブランチ切り替えで内容が戻ることを忘れない
**ルール**: hook の追加・変更は main マージ後まで「そのブランチ限定の設定」と認識する。  
**なぜ**: git 管理下のため、ブランチ切り替えで別ブランチの内容に上書きされる。  
**適用条件**: settings.json を変更したあとブランチ切り替えが発生するとき

## テスト

### コードとテストは必ず同じコミットで変える
**ルール**: ロジック・テキスト・embed を変えたら、アサートしているテストも同コミットに含める。  
**なぜ**: コードだけ変えてテストを後回しにすると、CI で失敗して余計な修正コミットが生まれる。  
**適用条件**: 既存コードを変更するとき常に

### テスト失敗は「PR前」ではなく「編集直後」に気づく
**ルール**: PostToolUse hook で `.py` 編集直後に pytest が自動実行される。結果を確認してから次の編集へ進む。  
**なぜ**: 修正コストは発見が遅いほど上がる。hook が最速の検知手段。  
**適用条件**: Python ファイルを編集するとき常に

## ドキュメント更新

### ボット名・embed 変更はコード・テスト・ドキュメントの3点セット
**ルール**: 1つ変えたら残り2つも同コミットで変える。`grep` で漏れを確認してからコミット。  
**なぜ**: 片方だけ変えると乖離が生じ、後から気づいて追加コミットが必要になる。  
**適用条件**: ボット名・embed テキスト・メッセージ文言を変更するとき

### 環境変数の追加・変更は3点セットで更新する
**ルール**: `.env.example`・`CLAUDE.md` の環境変数テーブル・workflow の `env:` ブロックを同時に更新。  
**なぜ**: 1つでも漏れると動作確認できない・CIが落ちる・ドキュメントが嘘になる。  
**適用条件**: 環境変数を追加・変更するとき

### 実装後は README.md・CLAUDE.md を必ずセットで更新する
**ルール**: 「実装完了」の定義にドキュメント更新を含める。実装だけで完了としない。  
**なぜ**: 「あとで直す」は高確率で忘れる。ドキュメント乖離はじわじわ蓄積する。  
**適用条件**: ファイル追加・構成変更・機能変更のとき

## Python環境・PYTHONPATH

### ディレクトリ移動後は PYTHONPATH・pytest.ini・workflow の3点を必ず確認する
**ルール**: ファイルを移動したら即座に `pytest.ini` → workflow → ローカル実行コマンドの順で確認する。  
**なぜ**: 移動だけして設定を直し忘れると CI が壊れる。ローカルで動いても CI で落ちる。  
**適用条件**: `src/` 以下のモジュールを移動するとき

### eventkun のローカル実行には PYTHONPATH の手動設定が必要
**ルール**: `PYTHONPATH="<abs>/src/reminkun;<abs>/src/eventkun" python src/eventkun/eventkun_main.py`（Windows は `;` 区切り）  
**なぜ**: モジュールが別ディレクトリをまたぐため、デフォルトでは `notifier` が見つからない。  
**適用条件**: ローカルで eventkun を単体実行するとき

## マルチエージェント

### 新ファイル作成＋複数ファイル変更の組み合わせはマルチエージェントで進める
**ルール**: 「新ファイル作成 + 複数ファイル変更」が発生したら `workflow.md` のマルチエージェント方針に従う。  
**なぜ**: 1エージェントで複数ファイルを同時変更するとコンテキストが混乱しやすく、見落としが増える。  
**適用条件**: workflow.md の「使う条件」に1つでも該当するとき
