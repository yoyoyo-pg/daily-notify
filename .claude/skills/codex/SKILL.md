# Codex

Codex CLIを使用してコードレビュー・分析を実行するスキル。

## 実行コマンド

```bash
codex exec --full-auto --sandbox <sandbox> --cd <project_directory> "<request>"
```

## sandbox の使い分け

| 用途 | sandbox オプション | 説明 |
|------|-------------------|------|
| レビュー・分析・調査 | `--sandbox read-only` | ファイルを読むだけ（安全） |
| コードの修正・生成 | `--sandbox workspace-write` | ファイルへの書き込みを許可 |

## プロンプトのルール

Codex に渡すリクエストには以下を必ず含めること：

1. **簡潔さの指示** — 出力が長くなりすぎると末尾が切れるため
   > 「回答は要点を箇条書きで簡潔にまとめてください。」

2. **自律実行の指示** — 確認待ちで止まらせないため
   > 「確認や質問は不要です。具体的な提案・修正案・コード例まで自主的に出力してください。」

3. **必要に応じてコンテキストを埋め込む** — Claude が持つ情報（エラー内容、読んだファイルの要点など）はプロンプトの冒頭に直接書くこと

## エラー時の対応

| エラー | 原因 | 対応 |
|--------|------|------|
| `Quota exceeded` | APIクレジット不足 | ユーザーに OpenAI の課金状況確認を依頼 |
| `Reconnecting...` 後に失敗 | 一時的な接続エラー | 1回だけ再実行。それでも失敗すればユーザーに報告 |

## ユースケース別の実行手順

### ブランチ差分レビュー（最頻出）

「codexにレビューさせて」「実装後にcodexでレビュー」と言われたらこのケース。

1. 変更ファイル一覧を取得する
   ```bash
   git diff origin/main..HEAD --name-only
   ```
2. 以下のプロンプトでCodexを実行する（`<変更ファイル一覧>` に実際のファイル名を埋め込む）
   ```bash
   codex exec --full-auto --sandbox read-only --cd <プロジェクトのフルパス> "$(cat <<'EOF'
   以下のファイルが origin/main から変更されています。
   <変更ファイル一覧>

   git diff origin/main で差分を確認し、以下の観点でレビューしてください：
   1. バグや論理的な誤り
   2. 意図しない副作用・見落とし
   3. 改善できる点

   問題なければ「LGTM」と一言だけ返してください。回答は要点を箇条書きで簡潔にまとめてください。確認や質問は不要です。具体的な指摘まで自主的に出力してください。
   EOF
   )"
   ```
3. 問題が指摘された場合は修正を促す。LGTM なら PR 作成に進む。

### バグ調査・設計相談・文言検討

1. 依頼内容が「レビュー・分析」か「修正・生成」かで sandbox を決める
2. 必要ならエラーや関連情報をプロンプトの冒頭に埋め込む
3. 簡潔さの指示と自律実行の指示を末尾に追加してコマンドを実行する
4. 結果をユーザーに報告する
